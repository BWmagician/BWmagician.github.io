作者 : BWmagician | 分类 : 关于hexo

准备工作

首先你要先确保安装了hexo的框架。

具体来说，你应该在根目录(有themes文件夹的那个目录)git bash并输入

$ hexo clean && hexo g && hexo s

后在对应的端口看到博客页面生成(一般是4000端口，推荐用谷歌浏览器看，edge加载会莫名卡顿，非常影响主题搭建)。

这一部分网上的教程很多，如果你难以抉择，根据官方文档的概述并完成建站即可。

本文章没有提到部署的操作。

在开始搭建主题之前，请确保hexo能正常工作。至少html和css和js语法要能懂。

关于作者可能是⑨这件事

由于作者对css和html和js的了解都不深入，并且在审美方面存在某种异变，因此整个主题与代码看起来会十分甚至⑨分简陋(非常便于理解)，如果某些大佬想尝试自己写一个主题，可以拿此主题当作一个参考例子。

并且，这篇博客写于完成主题的搭建之后，这篇博客实际上类似于文学上说的那种回忆录。因此有可能存在一些错误和偏差，如果你在实操中发现有什么不对劲的地方，请大胆的怀疑有可能是⑨犯了错误。

工作环境的选择(文本编辑器和ejs)

文本编辑器首先推荐vscode，vscode用来写这种中小型的project是非常合适的。

hexo默认提供了两种模板，分别是ejs和Nunjucks。请使用ejs吧 (因为没试过另外一种

ejs的作用就是能在你的html代码中插入一些js代码或其他东西，这会让你的主题搭建有一个质的飞跃。譬如说，hexo会给你准备的一些变量，其中包括了文章的标题。那么你该如何在写仅一份的html代码中展示不同文章的标题呢？

假设变量 page.title 就代表了当前页面的标题，我们可以写

<h1> <%= page.title %> </h1>

其中，<%= %>是ejs的语法，这个百分号尖括号包裹的内容遵循js语法，我们可以直接用js语法调用hexo为我们提供的变量。

那么当hexo把这个ejs语法的代码成真的html语法的代码时，他就会把真真切切的变量插入生成的html代码中，这对于码字的人来说可以一劳永逸了。

详细的语法可以参考ejs官方的文档

一般来说，我们只会用到<% %> <%- %> <%= %>三种百分尖括号的格式，具体情况下文会展开说，我们先把编辑器的环境配置好。

逆天缩进

如果你采用vscode默认的缩进格式，你会遇到一个致命的问题。vscode会把形如<% %>当成html中的标签，从而让本该并行的两行错开，比较要命的是当你保存的时候他还会自动给你缩进成扭曲的样子。

很遗憾，我并没有找到什么完美的解决方案，我的vscode在装了ejs的语法插件和Prettier插件后再也不会在保存时给我的代码缩进成鬼畜的样子了，但他也不会自动缩进了。

vscode的保存设置

还有，为了方便调试，建议把保存模式设置成失去聚焦就保存，这样我们改完代码在去浏览器点刷新之前就可以自动保存了。

hexo在干什么

在建主题之前，我们应该先了解一些hexo的一些逻辑，这有助于理解hexo的功能。

hexo的渲染和打包

首先hexo有一个渲染器，默认的应该是hexo-renderer-marked。这个渲染器能把一篇markdown格式的文章转化成了html的代码，感性理解一下就是把

# 这是一个一级标题

变成了(不一定对，但就是这个意思)

<h1> 这是一个一级标题 <h1>

然后hexo把你的markdown语法的文章都转换成html之后，他会打包一个文件夹，也就是你在根目录看到的public文件夹，这个文件夹的路径就对应了你生成的静态博客的url的路径。

举个例子，我们先按照他默认的主题渲染(先什么都别动)，在根目录git bash输入

$ hexo clean && hexo g && hexo s

我的hello-word.md渲染后的html代码(他默认给你生成的那篇文章)在public文件夹内的路径是这样的

public/2025/06/20/hello-world/index.html

对应的，这篇文章的url就是这样的

http://localhost:4000/2025/06/20/hello-world/

你可以打开一下public文件夹里的那个html文件，应该会长这个样子。

非常质朴，因为此时，hexo仅仅把layout.ejs的代码和post.ejs的代码和渲染出来的html代码搓到一起，直到浏览器自己把css代码和js代码加进去。public文件夹是为了方便在浏览器展示而生成的静态文件存放点。

layout.ejs 和 post.ejs 是什么及其具体功用会在下文提到，不必着急，至少我们已经知道了一个照着public文件夹路径找某篇文章的url的能力(你必须有一种直觉，就是我们刚开始写出来的东西很可能是一片空白)。

关于hexo的指令和变量

$ hexo clean 是删除public文件夹和缓存

$ hexo g 是生成public文件夹和缓存

$ hexo s是部署服务，也就是把这个页面接到你的4000端口上

这三个是最常用的指令并且经常是连着用的，于是我们可以直接三合一

$ hexo clean && hexo g && hexo s

如果你仔细阅读了hexo的官方文档，你一定会注意到hexo会给用户提供很多变量，hexo会解析你的文章和你文章和各种yaml配置并把他们变成变量。

如果你的hexo版本比较低或者你可以在根目录的_config.yml中添加了

cache: false

那么你甚至可以直接在根目录下看到db.json，也就是数据的缓存，他们会被解析成变量。

也就是说

$ hexo clean

是一个非常强大的指令，它能够清除缓存。换言之就是强刷新。在之后的搭建中，我们可能需要自己写一些函数，而这些函数的产生的数据可能就要存在缓存里。不幸的是这些函数难以一次性写到完美，当你要调试的时候，你不得不刷新错误的缓存，所以如果你遇到一些离奇的bug，不如直接试试这款强力的三合一指令吧

$ hexo clean && hexo g && hexo s

当然，如果你没有特别需求，直接hexo g 即可，因为hexo clean会消耗更多的时间。

helper

在某个ejs里调用一下

<% console.log(thisisaexample(-1)) %>

<% console.log(thisisaexample(0)) %>

可以看到⑨在输出结果之前重新用了三合一指令，这是为什么？因为⑨之前忘记加分号了，那么点浏览器的刷新就没有用了，这个报错会一直存在，需要清掉缓存。

⑨的理解是：scripts文件里的代码会在hexo启动server的非常早期进入那个数据缓存，并不是动态更新的。如果你不删除掉缓存或重启server，这段代码相当于没有更改过。

这个问题相当玄学，有的时候需要hexo clean之后才能解决，需要各位大佬自己调试

categories.ejs 分类, category.ejs 分类子页

需要手动添加categories页面。

回到根目录，git bash，

$ hexo new page categories

在source\categories\index.md中，加入layout这一行，表示对应到categories.ejs文件

css配置在 categories.css 和 category.css

写两个ejs的原因在第三部分(主题的搭建)说过了，就是做一些差分。

如果你从文章页面的具体分类链接进入分类子页，该文章的路径会自动打开折叠。

由于hexo自带的categories变量写的比较捞。那么只能自己写一个helper函数，在themes\bw\scripts\categries_folder.js中。考虑到html的折叠是details标签和summary标签的嵌套，可以先按照每篇category的路径把树建出来，然后在扫这颗树的时候，对于每一个节点用details标签夹住它的所有子树。对于分类子页，我们只要把某条路径的details标签改为<details open>即可。具体见代码注释。

这样做的好处是，可以在生成的html代码写入特定的类，然后我们只要在css里美化这些类即可。
